# 🚀 极简 AI 项目收尾与沉淀 Prompt 指南

在项目开发接近尾声时，为了能将代码资产、开发经验、踩坑记录以及部署说明彻底沉淀下来，我们总结出了以下 5 个维度的“收尾专用 Prompt”。
您可以直接将底下的 Prompt 复制发送给 AI，让其一键为您生成高质量的复盘文档并完成极为干脆俐落的项目收尾。

---

## 1. 撰写《项目代码复盘与总结》的 Prompt

**使用场景：** 项目基础功能开发完毕，需生成一份全局性的文字复盘文档，以便后续内部移交、团队分享或自我进阶复习。

**优化版 Prompt 模板：**
> “作为资深架构师和技术 Lead，请仔细提取并阅读我当前项目的所有最新代码与配置文件。请为我撰写一份详细而有深度的《项目代码复盘与总结.md》。
> 要求采用极简清晰的排版格式，并包含以下关键模块：
> 1. **全景概述**：用简练的语言描述项目初衷、核心功能亮点展示以及最终确认的具体技术栈组合。
> 2. **开发历程回顾**：对比开发初期的需求与最终的实现情况，列举开发过程中遭遇的最棘手的 3-4 个关键问题。
> 3. **核心难点与解法破解**：详细描述上述坑位产生的原因，以及我们最终采取的巧妙修复手段（例如前端的静态缓存顽疾是如何靠重命名 js 后缀与 HTML 断点强行清空的、跨越通讯死锁是如何被定位并靠 `flask_cors` 解决的等）。
> 4. **未来进阶展望**：提出当前系统能够在安全性扩展（如引入 JWT）、性能或功能拓展方面改进的 3-4 个高优建议。
> 5. **沉淀开发军规（通用 Rule）**：提炼出本次项目得出的放之四海而皆准的防御性编程法则（至少阐明 3 条，例如所有网络请求的前置防御与解除反馈机制）。”

## 2. 撰写《特定技术（飞书 API）避坑指南》的 Prompt

**使用场景：** 项目中深度集成了某个极具争议或“规矩森严”的第三方云平台（如飞书多维表格、Supabase、Notion），踩了无数隐藏的坑，需要专门沉淀一篇避坑排雷指南让后人不走弯路。

**优化版 Prompt 模板：**
> “基于我们在本项目开发中与【飞书多维表格 OpenAPI】艰难的对接痛点、报错处理和反复调试记录，请为我撰写一份高度具备实操性的《飞书开发注意事项与避坑指南.md》。
> 请完全从‘防坑与规避’的实战视角出发，包含以下核心要点：
> 1. **参数拦截潜规则**：警示搜索功能中的 `filter` 对象必须带有 `conjunction` 等毫无提醒的必填约束。
> 2. **返回值降级防沉降**：明确阐述原本应是单值的 Text 型字段变成了复杂的数组对象嵌套结构的成因，以及如果在 Python 不将其清洗为纯文本可能会导致前端引发什么样的致命灾难。
> 3. **隐匿的豪秒时间戳**：详述飞书可能返回的长串 Unix 毫秒级时间戳，警示研发人员如何及时将其格式化。
> 4. **无主键灾难与溯源**：解释平台无自建唯一键约束导致的数据频繁重复产生的后果，以及查询后再新增的人工屏障机制；另外强调把平台下发的随机 `record_id` 原路下传给前端作为后续操作凭证的必然性。
> 请务必包含对比鲜明的代码示例（导致报错的代码写法 vs 正确防护写法），言辞体现出极强的专业性和警示性。”

## 3. 生成《提示词 (Prompt) 提炼与复用指南》的 Prompt

**使用场景：** 在与 AI 协作实现系统后，想将本次沟通中摸索出的最佳提示词外挂提取出来，做为以后的同模态项目的“万能模板”。

**优化版 Prompt 模板：**
> “全面回顾我们在开发这个基于 Python + 飞书后端的极简共享日历时的反复沟通流程。我们在初期因为 Prompt 指令过于简短粗暴，导致陷入了无数次的本地跨域和‘页面死锁无响应’等折磨。
> 请为我撰写一份《提示词 (Prompt) 提炼与复用指南.md》。这不仅是一篇文章，更是我未来开发可以直接复用的‘超强外挂代码生成器模板’。
> 内容必须包含：
> 1. **反面教材展示 (Bad Case)**：列出最初那句寥寥数字的极简 Prompt 并一针见血地驳斥这会导致诸如缺乏‘防御交互约束’、‘忽略运行边界限制’等灾难原因。
> 2. **高阶模板本体 (Better Prompt)**：梳理并沉淀出一套包含了 [角色定位与大目标]、[详细技术栈约束]、[必须遵守的防御编程军规 Critical Rules (尤其是对异常拦截弹窗的极高要求)] 以及 [核心需求清单] 4 层结构的强悍模板，供人直接复制替换需求内容。
> 3. **通用性说明**：分析这套模板的防御思维精髓，并且阐明当未来需要让人工智能开发诸如‘报销单’或‘云端进销存账本’时，直接套用此框架将如何爆发出几十倍的编码成功率及容错率。”

## 4. 撰写开源规格 `README.md` 的 Prompt

**使用场景：** 项目要打包移交给他人、准备用于答辩、或是对外开源发布前，需要一个高逼格、能指导他人一步步跑起来的使用说明门面文档。

**优化版 Prompt 模板：**
> “你现在是一位知名的 GitHub 开源项目维护者，请为我目前的‘极简风格的高颜值团队共享日历’项目撰写一份达到开源天花板标准、极具美感的 `README.md`。
> 要求内容完整、图文并茂、模块清晰，至少包含以下骨架结构：
> 1. 🌟 **项目简介 (About)**：高昂有力的口号并用一两句话点明项目的定位与解决的痛点。
> 2. ✨ **核心特性 (Features)**：使用 Emoji 清晰直观地以列表维度提炼爆点（Apple风美学、前后端分离、无代码数据库云原生体验等）。
> 3. 🛠 **技术架构 (Tech Stack)**：分开罗列前端、后端框架、依赖项，并给出其各自在该项目中的职能角色说明。
> 4. 🚀 **快速开始 (Quick Start)**：**（最关键的核心项）** 详细描述一个小白如何一步步在本地将它完整启动起来。包含但不仅限于：
>    - 源代码克隆指南。
>    - Python 依赖激活配置（怎么使用 `requirements.txt`）。
>    - 飞书等云端密钥对接指南（去哪里获取密钥并填入项目中）。
>    - 怎么分别启动后端服务和前端 Live Server。
> 5. 📸 **界面预览与目录树 (Showcase & Tree)**：生成项目的源码目录骨架结构图，并预留图片引用的 markdown 占位符。
> 整个文档务必措辞精炼专业，排版使用多层级标题与代码块等使得视觉穿透力拉满。”

## 5. 推送/上传至 GitHub 仓库的代码收尾 Prompt

**使用场景：** 完全不懂 Git 命令行，不清楚哪些杂乱的文件必须被屏蔽，需要 AI 从头到尾保姆级给出上传当前本地项目所有的推库步骤、或直接代替敲击命令。

**优化版 Prompt 模板：**
> “我的共享日历项目代码此时都已妥当地停留在我的本地同级目录下。我想在项目收官时将其推送到我的 GitHub 远程个人仓库中持久化保存，但我对 Git 操作极其生疏。
> 请作为我的部署助理，提供以下极致的安全护航协助：
> 1. **核心文件过滤拦截过滤**：帮我检查并立即在根目录生成一个极为严苛的 `.gitignore` 保护文件。必须确保排除掉诸如 `__pycache__` 这种运行缓存、以及任何带包含有我们飞书自建真实 App ID 密钥的配置文件或 `.env` ，并过滤掉一些杂乱的纯测试文本（如 `清理.py`、`验证.txt`等）。确保只有纯净代码被传上去。
> 2. **零基础推送操作步进**：请给我提供在 Windows PowerShell 下最无脑、最详细的 Git 命令操作步骤流。从 `git init -b main` 初始化仓库、添加所有的安全文件、撰写一个极具极客范儿的首次 Commit 信息、到执行 `git remote add` 挂载云端地址，直至终端执行 `git push` 推送成功为止。
> 3. **排雷提示**：在所有命令旁边提供极简的中文含义。尤其叮嘱我如果在遇到远程代码库因为存在初始化的原生冲突（如远端带有 LICENSE 或默认 Readme 时）的合并解决，或是弹窗 Token 授权的强制要求时，我该怎么应对。
> *(补充：如果在拥有终端写入权限的大模型内，可直接追加下发指令：‘现在就直接调用 Terminal 帮我检查 Git 状态、建立清理完毕的 ignore 文件、执行暂存并完成首次提交，然后等待我告诉你 GitHub 仓库地址以便进行链接推送’。)*”
