# 共享日历项目 - Prompt (提示词) 提炼与复用指南

在本次使用 AI 辅助开发“基于 Python 与飞书多维表格后台的极简共享日历”时，我们总结了初版 Prompt 导致在本地调试阶段“前端静默卡死”和“数据格式报错”等问题的原因。
以下是经过改良、可直接复用及调整的高质量 Prompt 模板，旨在帮助您在未来类似的前后端分离开发中，一次性生成具备强大容错与极好体验的代码。

---

## ❌ 初版 Prompt (反面教材)

> 帮我写一个共享日历应用，要求手机可以访问，界面类似苹果风格，极简。
> 后端不用传统数据库，我要用 Python 调飞书的多维表格做存储。
> 前端用 HTML 和 Tailwind CSS。

**缺陷分析：**
1. **缺失本地调试边界限制**：没有声明前后端分离时势必带来的跨域和端口访问问题。
2. **缺乏防御性编程约定**：没有要求前端所有的业务请求（如点击登录、注册、拉取日程）必须要有明显的 UI 反馈（如报错弹窗）以及强制的异常捕获。
3. **未交待外部 API 的严格性**：未告知 AI 飞书接口存在的特殊数据结构（文本变多层列表、必需的 filter 配置等），导致生成的 Python 脚本过于乐观，一跑就报错。

## ✅ 改良后的高质量 Prompt (可直接复印 / 引用模板)

如果您下次需要开发类似系统，或者需要创建一个新的 H5 + 飞书/云端数据库应用，**请直接套用并修改以下 Prompt 结构**：

***

### 🧱 [角色与大目标]
作为一名资深的全栈工程师和高级 UX 设计师，请帮我开发一个**[填写类型，例：团队共享日历 / 任务看板]**应用。
要求具备极致简约的 Apple 风格（大字体、高对比度、平滑毛玻璃动画、微距点按反馈），并且必须做双端响应式适配（PC宽屏与移动端 H5 竖屏）。

### 🛠 [技术栈约束]
- **前端**：只使用纯 HTML5 + JavaScript (ES6+)，样式完全采用 Tailwind CSS（通过 CDN 引入），无需复杂的 Node.js/Vue/React 构建配置。
- **后端**：使用 Python (建议基于 Flask 框架) 构建轻量级服务端代码，作为中间人。
- **数据库生态**：不采用本地 SQLite/MySQL，全量业务数据基于调用 **「飞书开发平台 OpenAPI - 多维表格 Bitable」** 的记录读写接口来实现增删改查。

### 🚨 [必须遵守的开发约定 (Critical Rules)]

**1. 前端防御性交互约束 (非常重要)：**
所有涉及网络请求（Fetch）的按钮交互（例如点击提交 / 登录 / 注册 / 拉取数据）：
* 必须包裹严格的 `try...catch...finally` 块。
* 发起请求前立刻冻结（禁用）按钮并给以视觉 Feedback（比如文字变成 `提交中...`）。
* 在进入异常报错的 `catch` 流程，或者接口返回失败值时，**必须第一时间通过强提示弹窗 `alert(报错信息)` 通知用户**，同时**瞬间恢复按钮**的可用状态，**绝对禁止出现请求报错但界面静默无响应的“假死挂起”情况！**

**2. 跨域与网络通信保障：**
* 请显式地指明局域网开发环境的地址。Python 服务端代码中必须引入并应用 `CORS`，解决前端 Live Server 与 Python API 间的跨域拦截。
* 如果在浏览器由于缓存问题加载了旧版代码，请提供通过加后缀（如 `?v=timestamp`）避开静态文件缓存的指令注释。

**3. 飞书 Bitable OpenAPI 安全接入法则 (防报错)：**
* **条件搜索 `Search` 约束**：无论搜索条件多么简单（哪怕仅仅比对一个用户名），传给飞书的 `filter` JSON 中**必须包含 `conjunction: "and" (或 "or")` 参数**，严禁遗漏。
* **返回值剥离防护**：从飞书拉取的多维表格记录，其“文本型”字段很大可能是一个复杂的对象素组（例如 `[{"text": "张三"}]`），**禁止在后端直接将未清洗的数据传给前端**。请务必用代码在 Python 里做诸如 `val = arr[0].get("text") if isinstance(arr, list) else ""` 的强校验与提取。
* **业务排重意识**：飞书本身是不防重名注册的。编写新增（Create）接口逻辑前，必须先行调用 Search 接口比对唯一性字段，如果已存在，则报错阻断。

### 📋 [核心功能需求]
*(在这里逐条列出您期望这个系统能干什么，例如)*
1. 用户可以通过输入 用户名 和密码实现登录与注册，包含头像配色功能。
2. 登录后能够看到按当月日期排布的月视图日历，展示对应日期的所有团队排班或日程安排（用不同的色块深浅表示人员安排密度），支持上一月/下一月的快速翻页。
3. 点击指定日期可弹出浮层，展示该日期的详细待办事项列表；可在此进行新事物的添加，并且允许创建者删除自己创建的条目。

# 共享日历项目复盘与总结 (Project Retrospective)

## 1. 项目概述

**项目目标：**
开发一个高颜值的 H5 极简风（Apple 风格）团队共享日历应用，实现团队工作安排的可视化管理与实时同步。

**核心功能：**
*   **用户模块**：支持用户注册、登录，并自定义头像颜色与昵称。
*   **日历视图**：动态生成月度日历，支持日程数据读取；根据日程数量呈现不同的颜色深度（热力图标识）。
*   **日程管理**：日历点击查看详情，快速添加今日日程，允许用户主删除自己发布的日程。
*   **活动热力图**：专属的年度宏观视角，类似 GitHub 活动日历图。
*   **团队列表**：一目了然的团队成员与活跃状态展示。

**技术栈：**
*   **前端**：原生 HTML5 + JavaScript (ES6) + Tailwind CSS (通过 CDN 加载)。
*   **后端**：Python (基于 Flask 框架)。
*   **数据库**：基于飞书开放平台 OpenAPI 调用**飞书多维表格 (Bitable)** 作为远程数据库。

---

## 2. 开发过程回顾

### 2.1 提出的核心要求
1.  **UI 规范**：强烈的 Apple 极简美学，要求高对比度色彩、平滑动画和毛玻璃效果。
2.  **特定的存储要求**：不使用传统的本地结构化数据库（如 MySQL/SQLite），而是对接“飞书多维表格”，实现业务轻量化和跨系统的在线管理。
3.  **前后端分离但易于部署**：前端使用极简实现（无需 npm 打包加载），后端使用 Python 提供接口支撑。

### 2.2 遇到的问题与解决
1.  **“注册登录中”按钮长挂起被卡死的问题**
    *   **原因**：前端开发早期默认的 `API_BASE` 是按同源（生产环境）配置为空。但在真实的开发测试环境里（使用 Live Server 或直接按双击打开），前端与启动在 5000 端口的 Python 服务器属于跨域，导致 POST 请求直接发送到了无响应的地址，最终请求失败却没有抛出弹窗。
    *   **解决**：将后端的地址明确写入 JS 文件，且在 Flask 后端添加 `flask_cors` 跨域支持；对前端网络请求中的 `catch` 分支添加了强感知的弹窗警告（Alert）。
2.  **极度顽固的浏览器静态文件缓存**
    *   **原因**：修复了上述 JS 代码并添加 Alert 后，用户浏览器依然运行旧代码，导致点击无响应。
    *   **解决**：采取修改 JS 文件后缀名的方式（`app_v3.js`），并在 `index.html` 中同步更改引用路径，通过这种断点手段强制刷新浏览器缓存。
3.  **飞书 Open API 的数据格式校验极为严格**
    *   **原因**：飞书的 Bitable 搜索记录接口中，筛选条件 `filter` 的参数格式如果有轻微偏差（例如缺少了 `conjunction: "and"`），或者对字段类型的误判（多段文本返回数组），都会导致报错。
    *   **解决**：仔细查阅并补齐了飞书接口调用 SDK 相关的必须参数字典，对接口返回的数据结构（如 `[{"text": "val"}]` 形式）在后端进行了标准化的二次清洗。
4.  **测试脏数据的碰撞**
    *   **原因**：开发人员在调试阶段由于注册了与用户同名的临时测试记录，导致前端由于没有排重机制报出“密码错误”。
    *   **解决**：编写后台脚本 `cleanup.py` 清理垃圾数据。

## 3. Prompt 反思与改良思路

### 3.1 既有 Prompt 存在的盲区
刚开始下达的 Prompt，只注重了“最终结果（UI 与数据库选型）”，却忽视了“开发与调试环境”的特殊性：
*   没有提前声明如何运行前端页面，没有强制要求网络请求提供错误反馈墙（Error Boundary）。
*   没有声明飞书数据表结构的容错细节，导致中途花了很多时间排错。

### 3.2 下次可以使用的更好 Prompt (Better Prompt)
> “作为资深全栈工程师，请帮我开发一个基于 Python Flask + 飞书多维表格作为后端的团队共享日历应用。
> **关键约束：**
> 1. 前端使用 HTML+TailwindCDN 单文件构成，**请务必注意配置 CORS 跨域问题**，并写死开发环境下的后端默认接口（http://127.0.0.1:5000）。
> 2. 前端的所有 Fetch 通信必须包裹严密的 `try...catch`。在任何未响应、网络断开等情况发生时，必须有系统级的 `alert()` 弹窗提醒，并**立刻恢复按钮状态可用**，绝对不能产生静默卡死。
> 3. 对接飞书 API 时，请注意其返回的富文本值可能是 list 列表字典，请在 Python 端进行严格的提取与异常降级处理。
> 4. 界面需遵循极致苹果风设计：带有毛玻璃、自定义配色头像、呼吸灯与微距交互缩放。”

## 4. 未来的功能扩展与可以做的更好之处

1.  **更强大的安全鉴权（Tokens / JWT）**
    目前使用的是将用户信息序列化后存储在浏览器的 LocalStorage 中作为免密判断。为了具备商业可用性，可以在 Python 后端增加标准的 JSON Web Token (JWT) 下发机制。
2.  **权限的强力阻断**
    虽然前端屏蔽了针对他人日程的删除按钮，但在 API 接口层面（`/api/schedules/<id> DELETE`）还未强制验证请求的发起方是否属于该数据的创建人，未来需要在此进行加固。
3.  **消息推送（Webhook / 飞书机器人机制）**
    既然基础数据库使用了飞书，我们完全可以利用飞书的机器人生态。当有人增加或修改日程时，通过飞书群机器人直接推送一条日历更新消息到团队工作群。
4.  **工程化构建（Vite / Webpack）**
    为了彻底告别“重命名 js 文件破除缓存”这种原始方案，可以在未来的迭代中引入极简的工程部署（如 Vite），让每一次发布都能自动打上哈希（Hash）后缀，一劳永逸。

## 5. 项目沉淀的通用开发 Rule

*   **Rule 1: 防御性操作反馈**
    永远不要相信网络请求一定会成功。针对所有的“提交流程”，前端一定要做：`禁用按钮` -> `发起请求` -> `处理成功与否` -> `(在 finally 块中) 必定解除禁用按钮，并给出明确反馈`。
*   **Rule 2: 版本与缓存分离**
    对于外部引入的 `.js` 或 `.css` 静态文件，只要修改了，务必记得跟上版本号后缀（如 `?v=timestamp`）供客户端识别。
*   **Rule 3: 从数据底层解决重复**
    当系统允许用户手动输入“账号名”注册时，必须在写入数据库的逻辑之前加一道严格的“是否已存在检查”（唯一键约束或者业务查询拦截），防止脏数据叠加。

# 飞书多维表格 (Bitable) OpenAPI 避坑指南与开发注意事项

在将“飞书多维表格 (Bitable)”作为应用后段轻量级数据库进行集成调用的过程中，由于其严格的数据校验规则与特定的富文本嵌套结构，很容易导致新手遇到接口报错或数据解析异常。
本指南旨在总结本次“共享日历”项目的飞书对接经验，为了未来其它同样使用飞书的研发项目提供专业的复用指南。

---

## ⚠️ 1. 数据结构强校验陷阱：Filter 参数的“潜规则”

### 踩坑回顾：
在调用**搜寻记录接口 (`search`)**时，如果您仅仅提供了 `conditions`（查询条件）却未指定 **`conjunction` (条件关系)**，飞书后台不会默默通过，而是会无情地吐出 HTTP 400 报错并抛出：`field validation failed: filter.conjunction is required`。

### 避坑指南：
*   **凡是有 `filter` 的查询，必须配套 `conjunction`。** 即使您的筛选条件里只有唯一的一个规则 (比如只有一个字段要求匹配 `username == 178`)。
*   `conjunction` 的值必须且只能是：`"and"` （且）或 `"or"` （或）。

**正确的写法示范：**
```python
search_body = {
    "filter": {
        "conjunction": "and",  # 哪怕只有一个 condition，也必须写上
        "conditions": [
            {
                "field_name": "username",
                "operator": "is",
                "value": ["wangyang"]  # 这里的值也必须是一个 list 数组包裹
            }
        ]
    }
}
```

---

## ⚠️ 2. 单值字段变成“富文本数组”：返回数据的防沉降处理

### 踩坑回顾：
飞书多维表格里的“文本” (Text) 字段格式非常强大，它可以容纳富文本样式甚至 @ 人的内容。这就意味着，当你通过 API 获取刚才写入的例如 `"nickname": "王小明"` 的字段时，API 给你的返回值是一组列表结构，而非扁平字符串！
错误获取：`fields.get('nickname')` ->  拿到的是 `[{"type": "text", "text": "王小明"}]`，这给前端页面展示和二次判断带来了极大的灾难。

### 避坑指南：
*   **后端必须作为数据“护城河”进行类型降级。** 在向前端下发这些数据之前，您始终需要在服务端通过判断类型，从飞书复杂的“富文本对象列表”中抽取纯文本。

**正确的解析范式：**
```python
nickname = fields.get("nickname", "")
# 如果该值被飞书 API 以序列化格式返回（比如因为他是富文本当成数组处理）
if isinstance(nickname, list):
    # 抽取数组中第一个对象的 'text' 键，如果内容存在的话
    nickname = nickname[0].get("text", "") if nickname else ""
```

---

## ⚠️ 3. 飞书内建的时间戳问题：毫秒还是格式化字符串？

### 踩坑回顾：
飞书里的“日期”格式，在读写的时候常常造成混乱。如果你存储的字段被定义为“日期类型 (Type 5)”，飞书吐回前端时不仅没有格式化的时区字符串（如 `YYYY-MM-DD`），反而可能吐给你个令人困惑的长串整数——它是**带有毫秒的 UNIX 时间戳**，如果直接在不校验的情况下让前端使用，将渲染错误。

### 避坑指南：
*   **对于纯粹作为标识查询的字符串日期（如 “2026-03-15”）：强烈建议建表时建立为“多行文本 (Type 1)”而非“日期 (Type 5)”。** 这在搜索（例如搜索包含 `2026-03-` 的全月日历）和读写操作上要简单无数倍。
*   **如果坚持使用类型 5：** 在拉取数据时，如果拿到的是 `int/float`，必须在服务端先把它从毫秒时间戳转换出来。

**日期读取防护示例：**
```python
date_val = fields.get("date", "")
# 如果飞书返回的是时间戳
if isinstance(date_val, (int, float)):
    # 需要除以 1000 转回正常的秒级别再去格式化
    date_val = datetime.datetime.fromtimestamp(date_val / 1000).strftime('%Y-%m-%d')
```

## ⚠️ 4. 垃圾数据与主键管理

### 踩坑回顾：
多维表格没有天然的业务级“唯一键约束”在后台生效，如果接口（或本地调试期间）不进行防护拦截，随意 `POST` 可以创造出包含几十条一模一样的 `username` 的冗余数据。此外，每一行都有专属生成的随机且无法预测的 `record_id`，若你没有在拉取后暴露它并将其跟数据绑定，后续修改 / 删除请求就无从下手了。

### 避坑指南：
*   **查询后再新增：** 在 `Create` (注册/添加数据) 发生前，一定要预先使用带 `"conjunction"` 的 `search` API 探一探路：`if result.get("data", {}).get("total", 0) > 0: return fail`，以人工充当唯一键约束。
*   **下穿必备标识：** 下传给页面端的数据体中，必须把本记录原始且关键的 `item.get("record_id")` 也包裹进去。这是日后对该笔数据进行 `PUT` （修改）或 `DELETE`（废弃）追踪的唯一凭证。
