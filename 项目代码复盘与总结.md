# 共享日历项目复盘与总结 (Project Retrospective)

## 1. 项目概述

**项目目标：**
开发一个高颜值的 H5 极简风（Apple 风格）团队共享日历应用，实现团队工作安排的可视化管理与实时同步。

**核心功能：**
*   **用户模块**：支持用户注册、登录，并自定义头像颜色与昵称。
*   **日历视图**：动态生成月度日历，支持日程数据读取；根据日程数量呈现不同的颜色深度（热力图标识）。
*   **日程管理**：日历点击查看详情，快速添加今日日程，允许用户主删除自己发布的日程。
*   **活动热力图**：专属的年度宏观视角，类似 GitHub 活动日历图。
*   **团队列表**：一目了然的团队成员与活越状态展示。

**技术栈：**
*   **前端**：原生 HTML5 + JavaScript (ES6) + Tailwind CSS (通过 CDN 加载)。
*   **后端**：Python (基于 Flask 框架)。
*   **数据库**：基于飞书开放平台 OpenAPI 调用**飞书多维表格 (Bitable)** 作为远程数据库。

## 2. 开发过程回顾

### 2.1 提出的核心要求
1.  **UI 规范**：强烈的 Apple 极简美学，要求高对比度色彩、平滑动画和毛玻璃效果。
2.  **特定的存储要求**：不使用传统的本地结构化数据库（如 MySQL/SQLite），而是对接“飞书多维表格”，实现业务轻量化和跨系统的在线管理。
3.  **前后端分离但易于部署**：前端使用极简实现（无需 npm 打包加载），后端使用 Python 提供接口支撑。

### 2.2 遇到的问题与解决
1.  **“注册/登录中”按钮长挂起被卡死的问题**
    *   **原因**：前端开发早期默认的 `API_BASE` 是按同源（生产环境）配置为空。但在真实的开发测试环境里（使用 Live Server 或直接按双击打开），前端与启动在 5000 端口的 Python 服务器属于跨域，导致 POST 请求直接发送到了无响应的地址，最终请求失败却没有抛出弹窗。
    *   **解决**：将后端的地址明确写入 JS 文件，且对 Flask 后端添加 `flask_cors` 跨域支持；对前端网络请求中的 `catch` 分支添加了强感知的弹窗警告（Alert）。
2.  **极度顽固的浏览器静态文件缓存**
    *   **原因**：修复了上述 JS 代码并添加 Alert 后，用户浏览器依然运行旧代码，导致点击无响应。
    *   **解决**：采取修改 JS 文件后缀名的方式（`app_v3.js`），并在 `index.html` 中同步更改引用路径，通过这种断点手段强制刷新浏览器缓存。
3.  **飞书 Open API 的数据格式校验极为严格**
    *   **原因**：飞书的 Bitable 搜索记录接口中，筛选条件 `filter` 的参数格式如果有轻微偏差（例如缺少了 `conjunction: "and"`），或者对字段类型的误判（多宽文本返回数组），都会导致报错。
    *   **解决**：仔细查阅并补齐了飞书接口调用 SDK 相关的必须参数字典，对接口返回的数据结构（如 `[{"text": "val"}]` 形式）在后端进行了标准化的二次清洗。
4.  **测试脏数据的碰撞**
    *   **原因**：开发人员在调试阶段由于注册了与用户同名的临时测试记录，导致前端由于没有排重机制报出“密码错误”。
    *   **解决**：编写后台脚本 `cleanup.py` 清理垃圾数据。

## 3. Prompt 反思与改良思路

### 3.1 既有 Prompt 存在的盲区
刚开始下达的 Prompt，只注重了“最终结果（UI 与数据库选型）”，却忽视了“开发与调试环境”的特殊性：
*   没有提前声明如何运行前端页面，没有强制要求网络请求提供错误反馈墙（Error Boundary）。
*   没有声明飞书数据表结构的容错细节，导致中途花了很多时间排错。

### 3.2 下次可以使用的更优 Prompt (Better Prompt)
> “作为资深全栈工程师，请帮我开发一个基于 Python Flask + 飞书多维表格作为后端的团队共享日历应用。
> **关键约束：**
> 1. 前端使用 HTML+TailwindCDN 单文件构成，**请务必注意配置 CORS 跨域问题**，并写死开发环境下的后端默认接口（http://127.0.0.1:5000）。
> 2. 前端的所有 Fetch 通信必须包裹严密的 `try...catch`。在任何未响应、网络断开等情况发生时，必须有系统级的 `alert()` 弹窗提醒，并**立刻恢复按钮状态可用**，绝对不能产生静默卡死。
> 3. 对接飞书 API 时，请注意其返回的富文本值可能是 list 列表字典，请在 Python 端进行严格的提取与异常降级处理。
> 4. 界面需遵循极致苹果风设计：带有毛玻璃、自定义配色头像、呼吸灯与微距交互缩放。”

## 4. 未来的功能扩展与可以做的更好之处

1.  **更强大的安全鉴权（Tokens / JWT）**
    目前使用的是将用户信息序列化后存储在浏览器的 LocalStorage 中作为免密判断。为了具备商业可用性，可以为 Python 后端增加标准的 JSON Web Token (JWT) 下发机制。
2.  **权限的强力阻断**
    虽然前端屏蔽了针对他人日程的删除按钮，但在 API 接口层面（`/api/schedules/<id> DELETE`）还未强制验证请求的发起方是否属于该数据的创建人，未来需要在此进行加固。
3.  **消息推送（Webhook / 飞书机器人机制）**
    既然基础数据库使用了飞书，我们完全可以利用飞书的机器人生态。当有人增加或修改日程时，通过飞书群机器人直接推送一条日历更新消息到团队工作群。
4.  **工程化构建（Vite / Webpack）**
    为了彻底告别“重命名 js 文件破除缓存”这种原始方案，可以在未来的迭代中引入极简的工程部署（如 Vite），让每一次发布都能自动打上哈希（Hash）后缀，一劳永逸。

## 5. 项目沉淀的通用开发 Rule

*   **Rule 1: 防御性操作反馈**
    永远不要相信网络请求一定会成功。针对所有的“提交流程”，前端一定要做：`禁用按钮` -> `发起请求` -> `处理成功与否` -> `(在 finally 块中) 必定解除禁用按钮，并给出明确反馈`。
*   **Rule 2: 版本与缓存分离**
    对于外部引入的 `.js` 和 `.css` 静态文件，只要修改了，务必记得跟上版本号后缀（如 `?v=timestamp`）供客户端识别。
*   **Rule 3: 从数据底层解决重复**
    当系统允许用户手动输入“账号名”注册时，必须在写入数据库的逻辑之前加一道严格的“是否已存在检查”（唯一键约束或者业务查询拦截），防止脏数据叠加。
